# UseCoopInteraction

**Package:** `com.hypixel.hytale.builtin.adventure.farming.interactions`
**Type:** Transient Handler

## Definition

```java
// Signature
public class UseCoopInteraction extends SimpleBlockInteraction {
```

## Architecture & Concepts

The UseCoopInteraction class is a server-side, data-driven event handler that encapsulates the specific game logic for a player collecting produce from a chicken coop. It is a concrete implementation of the **Strategy Pattern**, extending the abstract SimpleBlockInteraction to provide specialized behavior.

Architecturally, this class sits within the server's Interaction Module. It is not a long-lived service but rather a stateless, single-purpose function object. Its existence is primarily defined by game asset configuration files, not hard-coded engine logic. This design allows game designers to attach complex behaviors like this one to any block type by referencing its codec, promoting a highly modular and data-driven world system.

Its core responsibility is to orchestrate a series of state changes within the world's Entity-Component-System (ECS) framework. It acts as a controller that reads from the player entity's state, modifies the target block entity's state (specifically the CoopBlock component), and updates the world's visual representation of that block.

The complete absence of logic in the `simulateInteractWithBlock` method is a critical design choice. It designates this action as **fully server-authoritative**, preventing client-side prediction and ensuring that the state of the coop's inventory is securely managed by the server, which is essential for preventing exploits.

### Lifecycle & Ownership

-   **Creation:** Instances of UseCoopInteraction are not created manually using the `new` keyword. They are deserialized from game asset definitions (e.g., JSON files) at server startup by a system utilizing the public static `CODEC` field. The server's Interaction Module then retrieves the appropriate, pre-instantiated handler when a player interacts with a corresponding block.
-   **Scope:** The scope of an instance is ephemeral, limited to the execution of a single interaction event. It is effectively a singleton from a functional perspective but transient in memory.
-   **Destruction:** As a stateless handler with no persistent references, the object is eligible for garbage collection immediately after the `interactWithBlock` method completes.

## Internal State & Concurrency

-   **State:** This class is **stateless and immutable**. All data required for its operation is passed as arguments to its methods. It does not contain any fields that change during its lifecycle.
-   **Thread Safety:** The class itself is inherently thread-safe due to its stateless nature. However, it operates on world-state objects like World, WorldChunk, and CommandBuffer, which are **not thread-safe**. The engine's architecture guarantees that all block interactions within a given region are processed serially on a single world-tick thread. Therefore, callers must not invoke this class's methods from multiple threads concurrently. Concurrency is managed by the calling Interaction Module, not by this class.

## Integration Patterns

### Standard Usage

A developer or designer does not invoke this class directly in code. Instead, it is specified within a block's asset definition file. The server's Interaction Module reads this configuration and dispatches interaction events to this handler automatically.

A conceptual asset configuration might look like this:

```json
// Example: my_coop_block.json
{
  "id": "my_coop_block",
  "interactions": [
    {
      "type": "Use",
      "handler": {
        // This 'codec' reference points to UseCoopInteraction.CODEC
        "codec": "hytale:use_coop_interaction"
      }
    }
  ]
}
```

When a player performs a "Use" interaction on `my_coop_block` in the world, the system automatically routes the event to an instance of UseCoopInteraction.

### Anti-Patterns (Do NOT do this)

-   **Direct Instantiation:** Never create an instance via `new UseCoopInteraction()`. The system relies on the `CODEC` field for deserialization and registration. Manual creation bypasses this entire data-driven architecture.
-   **Stateful Modification:** Do not add fields or state to this class. Its design contract is to be stateless. Introducing state will lead to unpredictable behavior and race conditions in a live server environment.
-   **Client-Side Logic:** Do not attempt to port this logic to the client. The `simulateInteractWithBlock` method is intentionally empty. The management of coop inventory is a sensitive, server-authoritative action.

## Data Pipeline

The flow of data for this interaction is unidirectional, originating from client input and resulting in a world state change that is then replicated back to clients.

> Flow:
> Client Input (Right-click on Coop) -> Network Packet -> Server Interaction Module -> **UseCoopInteraction.interactWithBlock** -> World State Mutation (CoopBlock component data is changed) -> Block State Update (Visual state changes from "Produce_Ready" to "default") -> Network Replication -> Client World Render Update
